# Accessibility Statement & VPAT

*ensemble v0.2-foundation*
*Last updated: 2026-05-17*

This document declares the current accessibility posture of the ensemble
spreadsheet SDK, based on automated audit + manual review against
[WCAG 2.2 Level AA](https://www.w3.org/TR/WCAG22/).

## Summary

| Component | WCAG 2.2 AA | Notes |
|---|---|---|
| Folder drawer + tree | ✓ pass | `role="tree"` + `role="treeitem"`, keyboard nav (↑↓ Enter F2 Delete), focus-visible ring |
| Drawer chrome | ✓ pass | `role="dialog"` + `aria-modal=true` + focus trap + return-focus-to-opener |
| Share dialog | ✓ pass | `<label>` for all controls, `aria-invalid`, sonner toasts use `role="alert"` |
| Version history panel | ✓ pass | `role="alert"` on error, three-state UI with `role="status"` |
| Activity timeline | ✓ pass | `role="alert"` on error |
| Toast (sonner) | ✓ pass | `richColors` mode includes `aria-live="polite"` |
| Univer canvas | partial | The Univer ribbon (`@univerjs/ui` 0.22) does not declare landmark `role="region"`; Univer's data grid is canvas-rendered and not screen-reader-accessible |
| Onboarding coach | ✓ pass | `role="dialog"`, `aria-labelledby` to title, skip button as first focusable |
| Top bar buttons | ✓ pass | All buttons have `aria-label` or visible text |
| Color contrast | partial | UI chrome verified at 4.5:1; cell data is user-content-driven |

## Known gaps (planned for v0.3+)

1. Univer canvas accessibility — Univer 0.22 paints all cells to a single
   `<canvas>`; screen readers cannot enumerate cells. The formula bar is
   keyboard-accessible.
2. Ribbon landmark role — Univer toolbar lacks `role="region"`.
3. High-contrast theme — `prefers-contrast: more` handler on roadmap.
4. RTL support — bi-di mirroring untested.
5. Screen reader testing — automated axe-core covers structural checks
   but live VoiceOver / NVDA verification pending.

## Testing methodology

- Automated: `@axe-core/playwright` against demo + folder drawer
  (`apps/demo/e2e/a11y.spec.ts`). Runs on every CI build.
- Manual: tab-key traversal; focus-ring visibility.

## VPAT 2.5 (abbreviated self-assessment)

Following the [VPAT 2.5 Rev 508 template](https://www.itic.org/policy/accessibility/vpat).
This is a self-assessment; hosts deploying for U.S. federal contracts should
commission an independent VPAT.

### Chapter 5: Software

| Criterion | Conformance | Remarks |
|---|---|---|
| 5.2 Activation of accessibility features | Supports | OS features flow through |
| 5.5 Operable parts | Supports | Keyboard-only operation |
| 5.7 Key repeat | Supports | Browser default respected |
| 5.9 Simultaneous user actions | Supports | All concurrent ops keyboard-reachable |

### Chapter 6: Web content

| WCAG 2.2 SC | Conformance | Remarks |
|---|---|---|
| 1.1.1 Non-text content | Partially Supports | UI icons have `aria-label`; canvas is content-driven |
| 1.3.1 Info and relationships | Supports | Semantic HTML5, ARIA tree/dialog/alert roles |
| 1.4.3 Contrast (Minimum) | Supports (chrome) | ensemble chrome meets 4.5:1 |
| 2.1.1 Keyboard | Supports | All chrome reachable via keyboard |
| 2.1.2 No keyboard trap | Supports | Drawer trap escapable via Esc |
| 2.4.3 Focus Order | Supports | DOM order matches visual order |
| 2.4.7 Focus Visible | Supports | `focus-visible` 2px ring |
| 3.2.1 On Focus | Supports | No unsolicited context changes |
| 3.3.2 Labels or Instructions | Supports | All form fields labeled |
| 4.1.2 Name, Role, Value | Supports | ARIA roles + names on custom widgets |
| 4.1.3 Status Messages | Supports | Toast `aria-live="polite"`; errors `role="alert"` |
